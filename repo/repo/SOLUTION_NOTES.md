# Описание решений

## Задача 1 — Парсер парковок Алматы

**Подход.** Выбрал официальное 2ГИС Catalog API вместо парсинга HTML-страниц. Причины: структурированные поля (вместимость, платность, часы работы) доступны без дополнительной обработки; не зависит от вёрстки сайта, которая может измениться; легально по ToS. Альтернативу (парсинг внутренних XHR-запросов сайта) рассматривал, но отклонил — она дала бы больше полей потенциально, но менее стабильна и рискованна с точки зрения условий использования.

**Главное ограничение — лимит demo-ключа: 1000 запросов/месяц** на Catalog (Places) API и отдельно 1000/месяц на Geocoder API. Наивный полный скан города с мелкой сеткой потребовал бы тысяч запросов (один тестовый запрос с радиусом 2 км в центре вернул 224 объекта при максимум 10 на страницу). Решение — двухэтапная адаптивная стратегия: (1) грубая сетка по всему городу (~88 ячеек, радиус 1.5 км, до 3 страниц на ячейку) — 98 запросов, 158 объектов; (2) рекурсивное разбиение только тех ячеек, где `total` превысил порог (это и есть реальные "горячие точки" — центр города) — ещё 124 запроса, итог 572 уникальных объекта. Суммарно 222 из 1000 запросов, с большим запасом.

**Дедупликация** — по уникальному `id` объекта в самом API, реализована на уровне сбора (множество `seen`). Проверил отдельно: 552 из 572 строк делят название с другими строками (например "Парковка №6047" встречается 35 раз) — оказалось, это не дубли, а отдельные физические точки одной городской платной парковочной зоны (разброс координат до 730 м). Чтобы это не выглядело как мусор, добавил поле `номер_зоны`, явно показывающее группировку.

**Что не успел/сознательно не сделал:**
- Точный адрес (улица/дом) — Catalog API не возвращает `address` для объектов `type=parking`. Использовал reverse-geocoding (Geocoder API) для получения района/микрорайона как приближения.
- Тариф — проверил поле `comment`/`address_comment` через `/items/byid` на выборке объектов, оно пустое для всех проверенных. Получить тариф можно было бы только парсингом HTML-карточек — отказался по ограничению времени.
- Тип объекта (ТЦ/БЦ/частная) и привязка "к какому объекту относится" — протестировал подход через поиск ближайших зданий (`point+radius`, поле `purpose_name`), но результат — технический классификатор 2ГИС ("Административное здание", "Сооружение"), а не нужная категоризация, плюс эвристика "ближайшее здание в радиусе" ненадёжна для уличных парковок без привязки. Решил не тратить ~570 запросов (больше половины оставшегося лимита) на поле с сомнительным качеством.

## Задача 2 — Рассыльщик WhatsApp

**Подход.** Для live-режима выбрал автоматизацию WhatsApp Web (`pywhatkit`) вместо официального WhatsApp Business Cloud API — последний требует регистрацию и верификацию Meta Business аккаунта, что рискованно не успеть за 5 дней и не контролируется напрямую. Для демонстрации принципа (dry-run/live, обработка ошибок) разница в используемом транспорте не принципиальна.

**Защита от дисквалификации.** dry-run — режим по умолчанию, не делает сетевых вызовов вообще. Live физически ограничен в коде: требует отдельный whitelist-файл максимум на 2 записи, основная таблица (572 строки) никогда не используется как источник адресатов в live — даже ошибочный запуск `--live` с большим input-файлом не приведёт к массовой рассылке. Номера в демо-таблице для dry-run — синтетические плейсхолдеры (`+7700000001...`), не реальные.

**Обработка ошибок.** Валидация формата номера (regex на `+<код страны><номер>`), проверка пустых полей, отдельная обработка отсутствующего/некорректного whitelist-файла для live — всё протестировано на граничных случаях, программа не падает, продолжает обработку остальных строк.
